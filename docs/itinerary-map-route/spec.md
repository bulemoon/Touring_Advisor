# Spec — 攻略路线地图联动

> 状态：Done
> 日期：2026-05-17

## 背景

用户在 AI 对话中输入城市、地方或旅行偏好后，系统需要生成可执行的旅游攻略。攻略不应只停留在文本层面，还需要在地图上同步展示推荐地点、合理游览顺序和路线路径。

当前项目已有 Leaflet + 高德地图底图，以及 `route_planner` 路线规划能力。本功能目标是打通“AI 攻略生成 → 结构化路线 → 地图展示”的完整链路。

## 目标

- 用户通过聊天获得城市或地点攻略后，地图自动显示攻略地点。
- 系统根据地点位置合理安排游览顺序，减少折返。
- 地图展示编号地点 marker、路线 polyline、地点弹窗信息。
- 聊天文本与地图路线保持同步。
- 兼容现有路线卡片选择逻辑。

## 非目标

- 暂不实现完整多城市远程 POI 检索。
- 暂不实现真实多日分段地图图层切换。
- 暂不实现用户手动拖拽调整路线。
- 暂不引入新的地图 SDK。
- 暂不改数据库结构。

## 用户故事

### 用户故事 1：聊天生成攻略后看地图

用户输入：

> 帮我安排一个鼓浪屿半日游

系统返回文字攻略，并在地图上显示 3-5 个地点，按推荐游览顺序编号，同时画出路径。

### 用户故事 2：用户指定交通偏好

用户输入：

> 帮我安排上海临港一日游，适合自驾

系统识别自驾偏好，后端按 driving 模式生成路线，地图展示自驾路径。

### 用户故事 3：路线卡片和聊天路线共存

用户点击路线推荐卡片时，地图显示该路线。
用户随后在聊天中生成新攻略时，地图切换为聊天生成的路线。

## 功能需求

### 1. 攻略意图识别

系统需要识别以下关键词：

- 攻略
- 路线
- 行程
- 旅游
- 旅行
- 景点
- 打卡
- 半日游
- 一日游
- 两日游
- 三日游
- 怎么玩
- 去哪玩

命中后，聊天接口尝试生成地图路线。

### 2. 地点来源

MVP 阶段优先使用用户当前位置周边 POI。

输入城市或地点名时，可提取城市/区域名作为 prompt 上下文，但不保证跨城市检索准确性。

后续版本应升级为：

- 高德关键字搜索
- 城市限定 POI 检索
- 目的地中心点定位
- 再基于中心点生成路线

### 3. 路线结构

后端聊天接口返回：

```json
{
  "type": "message",
  "content": "文字攻略",
  "route_plan": {
    "title": "路线标题",
    "summary": "整体建议",
    "coord_system": "gcj02",
    "mode": "walking",
    "optimized": true,
    "start": {},
    "days": [],
    "stops": [],
    "polyline": [],
    "distance_m": 0,
    "travel_duration_sec": 0
  }
}
```

### 4. 地图展示

地图需要展示：

- 编号 marker
- polyline 路径
- 自动 fitBounds 到完整路线
- popup 展示：
  - 顺序
  - 地点名
  - 地址
  - 建议停留时长
  - 推荐理由

### 5. 前端状态

聊天 store 保存当前生成路线：

```ts
currentRoutePlan: RoutePlan | null
```

`App.vue` 持有全局 `routePlan`，传入 `MapArea.vue`。

路线来源包括：

- `TourSection` 路线卡片
- `ChatSection` AI 聊天

后触发者覆盖当前地图路线。

## 接口契约

### POST /api/chat

请求：

```json
{
  "message": "帮我安排一个半日游",
  "lat": 31.2304,
  "lng": 121.4737
}
```

响应：

```json
{
  "type": "message",
  "content": "推荐攻略文本",
  "route_plan": {}
}
```

如果无法生成路线：

```json
{
  "type": "message",
  "content": "普通聊天回复",
  "route_plan": null
}
```

## 验收标准

- 输入攻略类请求后，聊天返回正常文本。
- 如果周边 POI 足够，地图展示至少 2 个地点。
- 地点 marker 编号与后端 `sort_order` 一致。
- 地图显示路径线。
- 点击 marker 可看到地点信息。
- 前端构建通过。
- 后端 Python 编译通过。

## 风险

- Kimi 返回非 JSON 时会导致无法生成结构化路线。
- 用户输入远程城市时，当前 MVP 仍可能使用当前位置周边 POI。
- 高德 Key 缺失时只能返回直线 polyline。
- POI 名称匹配不稳定时，可能少量地点无法映射坐标。

## 后续增强

- 使用高德关键字搜索支持指定城市。
- 增加路线来源标识：聊天生成 / 推荐卡片。
- 增加多日路线切换 Tab。
- 保存用户生成过的路线。
- 支持手动重新排序和重新规划。
